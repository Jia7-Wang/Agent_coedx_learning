## 2. 什么是 Tool Calling

Tool Calling 可以简单理解成：

`模型不会直接假装自己什么都知道，而是在需要时请求程序去调用某个工具。`

例如：

- 查天气
- 查日程
- 查数据库
- 查订单
- 查知识库

这些事情很多都不是模型自己“记住”的，而是需要借助工具。

---

## 3. 今天这个示例会做什么

今天我们不接真实数据库，也不接真实天气 API。  
我们先用一个本地“假工具”来学习流程。

这个工具的名字是：

`get_learning_plan_by_level`

它的作用是：

- 输入学习者等级
- 返回对应的学习建议

例如：

- `beginner` -> 新手学习建议
- `intermediate` -> 中级学习建议
- `advanced` -> 高级学习建议

这样你可以专注理解 Tool Calling 本身，而不会被外部 API 干扰。

---

## 4. Tool Calling 的最小闭环

今天你一定要看懂这 4 步：

1. 程序把工具清单告诉模型
2. 模型判断是否要调用工具
3. 程序执行工具并拿到结果
4. 程序把工具结果再发回模型，让模型生成最终答案

这就是最基本的 Agent 执行闭环。

---

## 5. 为什么这一步特别重要

因为从 Day 3 开始，模型不再只是“回答问题”，而是开始参与“决策流程”：

- 要不要查工具
- 查哪个工具
- 工具参数是什么
- 查完之后如何组织最终回答

这就是 Agent 和普通聊天机器人的分水岭。

## 8. 今天你重点要看懂什么

### 8.1 工具不是普通函数那么简单

虽然工具最终会落到 Python 函数上，但对模型来说，它首先是一个“可用能力声明”。

也就是说，你要先告诉模型：

- 工具叫什么
- 这个工具做什么
- 需要什么参数

### 8.2 模型不会真的执行工具

模型只会：

- 提出调用请求
- 给出工具名
- 给出参数

真正执行工具的是你的 Python 程序。

### 8.3 程序要负责兜底

即使模型发起了工具调用，也不是说一切都安全了。

程序还要负责：

- 校验参数
- 执行工具
- 处理异常
- 把结果返回模型

## 10. 常见疑问

### 10.1 为什么不用真实工具

因为今天的重点是：

`先理解流程，再接真实世界`

如果一开始就接天气 API、数据库、权限系统，很容易被细节淹没。

### 10.2 中转 API 会不会不兼容

有可能。

所以今天这个脚本会先尝试标准工具调用方式，并把结果打印出来。  
如果你的中转不支持，我们就退一步，用“结构化输出模拟工具调用”的方式继续学原理。
