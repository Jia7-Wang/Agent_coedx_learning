# Agent 开发工程师学习路线图

## 1. 文档目的

这份文档是为零基础学习者准备的 Agent 开发入门手册。

目标不是让你只会一些名词，而是帮助你一步步具备独立开发 Agent 系统的能力，并且同步掌握公司项目中的基本工程协作常识，最终成长为一名可以做项目、能协作、会调试、懂评测的 Agent 开发工程师。

你可以把 Agent 理解为：

`能理解任务、会调用工具、能在状态和约束下完成工作的 AI 系统`

一个 Agent 系统最核心的 5 个组成部分是：

- `Model`：模型本身，负责理解和生成。
- `Instructions`：系统指令，定义角色、边界和目标。
- `Tools`：工具，让模型能够访问外部世界。
- `State`：状态，记录上下文、会话信息和流程进度。
- `Evals`：评测，用来验证系统是否真的可靠。

## 2. 先建立正确认知

### 2.1 Agent 不是“会聊天的模型”

普通聊天模型的重点是“回答问题”。

Agent 的重点是：

- 理解任务
- 判断下一步该做什么
- 决定是否调用工具
- 处理工具返回结果
- 在必要时继续下一步
- 最终给出结果或交给人工

所以 Agent 更像“会思考流程的程序”，而不是单纯“更聪明的对话机器人”。

### 2.2 Agent 开发的本质

Agent 开发最核心的能力，不是让模型更神秘，而是：

`把模型接入真实世界工具，并在可控流程中稳定完成任务`

这意味着你未来真正需要练的是：

- 写清晰的提示词
- 定义稳定的工具接口
- 设计可靠的工作流
- 处理错误和权限边界
- 做评测和观察日志

### 2.3 不要一开始就追多 Agent

更稳妥的顺序应该是：

1. 先学单模型调用
2. 再学工具调用
3. 再学单 Agent
4. 再学状态和工作流
5. 最后再学多 Agent 编排

---

## 3. 你要成为怎样的 Agent 工程师

一名合格的 Agent 开发工程师，通常要具备下面这些能力：

- 能用 Python 独立开发 Agent 原型
- 能接 API、数据库、搜索、文件系统等外部工具
- 能设计有状态的执行流程
- 能做知识库问答与 RAG
- 能做基础评测、日志追踪和问题排查
- 能处理安全、权限、重试、超时等生产问题
- 能把 Agent 封装成接口或应用并部署上线
- 能进入团队项目并进行基础协作开发

你的目标不是“会一个框架”，而是“具备完整解决问题的能力”。

---

## 4. 学习 Agent 时，必须同步补齐的公司项目常识

如果你的目标是学完后尽快上手公司项目，那么你的学习不能只有模型和框架，还要同步补齐工程协作能力。

很多初学者会出现这种情况：

- 会调用模型
- 会写 demo
- 会做简单 Agent
- 但不会看项目结构
- 不会拉代码
- 不会开分支
- 不会写 commit
- 不会排查日志
- 不会跑服务

这样即使你懂一些 Agent，也很难快速进入真实团队开发。

所以从现在开始，你的学习路径应该是两条线并行：

### 线路 A：Agent 能力线

- 模型调用
- 工具调用
- 工作流
- RAG
- 多 Agent
- 评测与上线

### 线路 B：工程协作线

- Git
- 命令行
- 编辑器使用
- 项目结构理解
- 环境配置
- 日志与调试
- 测试
- Docker
- 接口联调
- 基础部署

真正能快速入职的人，往往不是只会 Agent 的人，而是：

`既会做 Agent，又能进入团队工程流程的人`

## 5. 推荐学习路线总览

建议你按下面顺序学习：

1. `Python 基础`
2. `HTTP / API / JSON 基础`
3. `Git / 命令行 / VS Code 基础`
4. `LLM 基础与 Prompt 基础`
5. `结构化输出`
6. `Function Calling / Tool Calling`
7. `单 Agent 开发`
8. `RAG 与知识库`
9. `状态管理与工作流`
10. `多 Agent 编排`
11. `MCP`
12. `评测、Tracing、Guardrails`
13. `部署与生产化`
14. `Linux / Docker / 基础上线`

如果你完全零基础，不建议跳步。

## 6. 分阶段学习计划

## 第 1 阶段：基础准备期

### 目标

掌握最基础的编程和 API 调用能力，为后续 Agent 开发做准备。

- Python 基础语法

- 文件读写
- `requests`
- `async/await` 基础概念
- 环境变量
- JSON 数据结构
- HTTP 请求基础
- REST API 基础概念
- Git 基础命令
- 命令行基础
- VS Code 基础使用

### 这一阶段要穿插掌握的公司常识

#### Git 入门

你至少要会下面这些命令和概念：

- `git init`
- `git clone`
- `git status`
- `git add`
- `git commit`
- `git log`
- `git pull`
- `git push`
- 分支是什么
- 为什么不能直接在主分支乱改

#### 命令行基础

你至少要会：

- 查看当前目录
- 切换目录
- 查看文件
- 新建文件夹
- 运行 Python 脚本
- 安装依赖
- 查看环境变量

#### 编辑器基础

你至少要会：

- 用 VS Code 打开项目
- 搜索全局代码
- 查看终端输出
- 安装 Python 扩展
- 运行脚本
- 看报错位置

### 你必须能做到

- 能写一个 Python 脚本
- 能读取 `.env` 或环境变量
- 能调用第三方 API
- 能解析 JSON 响应
- 能做简单错误处理
- 能用 Git 提交一次代码
- 能在命令行中进入项目并运行脚本

### 建议输出

- 一个调用天气 API 的脚本
- 一个读取本地文件并总结内容的脚本
- 一个接受用户输入并调用模型的脚本

### 学完标志

你能不依赖别人，独立写出一个 `Python + API` 小程序。

## 第 2 阶段：LLM 调用与结构化输出

### 目标

理解模型接口，学会让模型稳定地产生可用结果，而不是只会自然语言对话。

### 你要学什么

- Chat Completions / Responses 类接口的基本思想
- `system`、`user`、`assistant` 消息角色
- temperature、max tokens 等基本参数
- 结构化输出
- JSON schema 基本概念
- Prompt 设计基础
- Debug 基础
- 日志打印基础

### 这一阶段要穿插掌握的公司常识

#### Debug 思维

公司项目里，很多时间不是在写代码，而是在排查问题。你要开始养成下面的习惯：

- 看完整报错，不只看最后一行
- 先确认输入、输出、环境变量
- 在关键节点打印日志
- 把问题缩小到最小可复现范围

#### 日志意识

你要知道：

- 为什么不能只靠肉眼看结果
- 为什么线上问题需要日志
- 什么地方适合打印日志
- 什么信息不应该写进日志，例如敏感密钥

### 你必须能做到

- 让模型按指定格式返回结果
- 让模型输出固定字段的 JSON
- 处理模型不按格式返回的异常情况
- 能通过报错和日志定位简单问题

### 建议输出

- 文本分类器
- 关键词提取器
- 简历信息结构化提取器

### 学完标志

你已经不只是“问模型问题”，而是开始“把模型当系统组件使用”。

---

## 第 3 阶段：Tool Calling 与单 Agent

### 目标

真正进入 Agent 开发的核心阶段，理解模型如何借助工具完成任务。

### 你要学什么

- Function Calling / Tool Calling
- 工具 schema 设计
- 参数校验
- 工具执行闭环
- 工具异常处理
- 单 Agent 基础架构
- 分支开发基础
- Code Review 基础意识

### Agent 的最小执行闭环

1. 给模型任务和可用工具
2. 模型判断是否需要调用工具
3. 你的程序执行工具
4. 将工具结果传回模型
5. 模型继续决定下一步
6. 输出最终答案

### 这一阶段要穿插掌握的公司常识

#### Git 分支协作

开始进入“像团队一样开发”的阶段。你应该学会：

- `git checkout -b <branch-name>` 或 `git switch -c <branch-name>`
- 为什么一个需求通常对应一个分支
- 提交前为什么要先看 `git diff`
- 提交信息为什么要写清楚

#### Commit 习惯

建议你从现在开始练这几件事：

- 一次 commit 只做一类改动
- commit message 尽量描述清楚目的
- 不要把临时代码、测试垃圾一起提交
- 提交前先运行最基本的检查

#### Code Review 基础意识

你未来进公司后，代码大概率会被 review。现在就要开始关注：

- 改动是否容易看懂
- 命名是否清晰
- 有没有明显边界条件遗漏
- 异常处理是否合理

### 你必须能做到

- 自己定义一个工具
- 把工具注册给模型
- 执行模型发起的工具调用
- 返回工具结果并得到最终回复
- 能在自己的分支上完成开发并提交代码

### 建议输出

- 天气查询 Agent
- 日程查询 Agent
- 文件分析 Agent
- “问模型要不要查数据库”的 Agent

### 学完标志

你已经真正写出了一个最基本的 Agent 系统。

---

## 第 4 阶段：RAG 与知识库 Agent

### 目标

让 Agent 能基于外部知识回答问题，而不是只依赖模型参数记忆。

### 你要学什么

- RAG 基本原理
- 文档切分
- 向量化与 embedding
- 召回与重排
- 引用来源
- 知识库问答的常见错误
- 项目目录结构理解
- 配置文件管理

### 你要理解的关键区别

- `LLM` 解决“怎么说”
- `RAG` 解决“知道什么”
- `Agent` 解决“下一步做什么”

### 这一阶段要穿插掌握的公司常识

#### 项目结构理解

你需要开始习惯看到一个项目时先判断：

- 入口文件在哪里
- 配置文件在哪里
- 依赖写在哪里
- 接口层、服务层、工具层如何划分
- 日志和测试目录在哪里

#### 配置管理

你要理解这些常见文件的作用：

- `.env`
- `requirements.txt`
- `pyproject.toml`
- `README.md`
- `docker-compose.yml`

### 你必须能做到

- 把 PDF、Markdown 或网页内容变成知识库
- 让 Agent 先检索，再回答
- 给出引用来源
- 避免无依据乱答
- 能看懂一个中小型 Python 项目的目录结构

### 建议输出

- 企业知识库问答助手
- 技术文档助手
- 学习资料问答助手

### 学完标志

你已经能做出一个比较像真实业务产品的 Agent。

---

## 第 5 阶段：状态管理与工作流

### 目标

让 Agent 不只是“一问一答”，而是能处理多步骤任务。

### 你要学什么

- session 和上下文管理
- memory 与 state 的区别
- 多步骤任务拆解
- 工作流编排
- 失败重试
- 超时控制
- 人工确认节点
- 测试基础
- 接口联调基础

### 你要建立的正确认知

“记忆”不是魔法。

大部分所谓长期记忆，本质上只是：

- 保存历史消息
- 提取用户偏好
- 存储任务状态
- 做检索增强

### 这一阶段要穿插掌握的公司常识

#### 测试基础

你至少要接触：

- 单元测试是什么
- 集成测试是什么
- 为什么改完代码要验证
- 如何为核心函数写最基础测试

#### 接口联调

Agent 项目经常要跟后端、前端、数据库或第三方服务联调。你要开始理解：

- 请求参数和响应格式
- 状态码
- 超时和重试
- mock 数据的用途

### 你必须能做到

- 跟踪任务进行到哪一步
- 在不同步骤中保留必要状态
- 在失败时能恢复或重试
- 在高风险操作前要求确认
- 能为核心逻辑写基础测试

### 建议输出

- 表单填写 Agent
- 工单处理 Agent
- 订单查询与退款 Agent

### 学完标志

你已经开始具备“业务自动化工程师”的能力，而不只是“AI demo 开发者”。

---

## 第 6 阶段：多 Agent 编排

### 目标

学会在复杂系统中拆分职责，让多个 Agent 或角色协作完成任务。

### 你要学什么

- Manager 模式
- Handoff 模式
- 专家 Agent 设计
- 路由策略
- 协作边界
- 多 Agent 的复杂度控制
- 文档协作基础
- 需求拆解基础

### 两种常见模式

#### 1. Manager Pattern

一个主 Agent 统筹全局，调用不同的专用 Agent 完成子任务。

#### 2. Handoff Pattern

根据任务类型，把用户对话转交给更合适的专用 Agent。

### 这一阶段要穿插掌握的公司常识

#### 写文档的习惯

在公司里，能写清楚文档的人推进效率会高很多。你要练习写：

- 功能说明
- 接口说明
- 配置说明
- 使用说明
- 问题复盘

#### 需求拆解

你要开始练习把一个模糊需求拆成：

- 用户要什么
- 系统要做什么
- 需要哪些工具
- 哪些地方高风险
- 哪些部分可以先做 demo

### 你必须能做到

- 判断什么时候需要多个 Agent
- 避免无意义拆分
- 控制上下文传递成本
- 保证交接过程可追踪
- 能把一个需求拆成几个清晰子任务

### 建议输出

- 客服分流系统
- 研究助理系统
- 内容生产协作系统

### 学完标志

你已经能设计较复杂的 Agent 架构。

## 第 7 阶段：MCP、评测与生产化

### 目标

把 Agent 从“能跑”提升到“可靠、可维护、可上线”。

### 你要学什么

- MCP 基本思想
- 标准化工具接入
- Tracing
- Evals
- Guardrails
- 权限控制
- 监控与日志
- 成本与延迟优化
- 部署与服务化
- Linux 基础
- Docker 基础
- 线上问题排查意识

### 这一阶段要穿插掌握的公司常识

#### Linux 基础

很多公司项目最终会跑在 Linux 环境中，所以你至少要会：

- 查看文件
- 查看日志
- 查端口
- 查进程
- 杀进程
- 查看环境变量

#### Docker 基础

你至少要理解：

- Docker 是做什么的
- 镜像和容器的区别
- 如何启动一个基础服务
- 为什么团队喜欢用 Docker 保证环境一致

#### 线上排障意识

你要开始理解真实项目中的问题往往来自：

- 配置错误
- 权限问题
- 网络超时
- 工具返回异常
- 上下文超长
- 成本过高或速度过慢

### 你必须能做到

- 看 trace 定位问题
- 设计测试集评估 Agent
- 控制高风险工具调用
- 对关键操作增加审批
- 让 Agent 以 API 或 Web 服务形式运行
- 能在基础 Linux / Docker 环境下运行项目

### 建议输出

- 带日志追踪的业务 Agent
- 带评测脚本的问答 Agent
- 带权限控制的执行型 Agent

### 学完标志

你已经具备初级到中级 Agent 工程师的工程化能力。

## 7. 推荐技术栈

对于零基础到入门阶段，我推荐下面这套组合：

- 编程语言：`Python`
- Web 框架：`FastAPI`
- Agent 入门框架：`OpenAI Agents SDK`
- 工作流编排：`LangGraph`
- 工具协议：`MCP`
- 数据库：`PostgreSQL`
- 缓存：`Redis`
- 向量检索：任选一个成熟方案即可，先别纠结品牌
- 部署：`Docker`
- 协作工具：`Git + GitHub / GitLab`
- 编辑器：`VS Code`

### 为什么这样选

- Python 生态最成熟，示例最多，适合初学者
- FastAPI 很适合做接口服务
- OpenAI Agents SDK 更适合理解 Agent 的基本构成
- LangGraph 适合复杂流程和有状态任务
- MCP 会越来越重要，值得尽早建立概念

## 11. 新手最容易踩的坑

### 坑 1：过早追求多 Agent

很多问题用单 Agent 加几个工具就够了。多 Agent 会显著增加复杂度、调试难度和上下文成本。

### 坑 2：只会写 Prompt，不会做工程

真正上线的 Agent，难点常常不在 Prompt，而在：

- 工具失败
- 超时
- 重试
- 权限控制
- 数据格式不稳定

### 坑 3：迷信“记忆”

很多所谓记忆系统，其实只是历史消息存储和检索。不要把它神化。

### 坑 4：不做评测

没有评测，你就不知道 Agent 是真的好，还是只是偶尔表现不错。

### 坑 5：只看教程，不做项目

Agent 开发是非常强调实践的领域。光看不练，进步会很慢。

### 坑 6：会写 demo，但不会进公司项目

如果你完全不补 Git、调试、测试、Docker、日志这些能力，那么即使会做 Agent demo，也很难直接上手团队项目。

## 12. 学习方法建议

### 12.1 每学一个概念，就做一个最小 demo

不要等“全学完再做项目”。

最好的节奏是：

- 今天学结构化输出，今天就做提取器
- 今天学 tool calling，今天就做天气 Agent
- 今天学 RAG，今天就做文档问答

### 12.2 每个阶段都保留作品

建议你把所有练习都放进一个代码仓库或学习目录，形成自己的作品集。

同时建议你从第一天就开始建立下面这些工程习惯：

- 每个项目写 `README`
- 每次改动前先开分支
- 每完成一个小功能就提交一次
- 每个项目保留运行方式和依赖说明

### 12.3 养成写复盘的习惯

每做完一个项目，记录这 4 件事：

- 这个 Agent 解决什么问题
- 用到了哪些工具
- 出过哪些 bug
- 下次怎么优化

### 12.4 不要过度追框架

框架会变，但底层能力变化没那么快。

你要优先建立的是：

- 模型调用能力
- 工具设计能力
- 工作流设计能力
- 工程化能力
