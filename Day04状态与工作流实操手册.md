---

## 2. 什么是状态

状态可以先简单理解成：

`程序为了继续完成任务，必须记住的中间信息`

例如一个多步骤任务里，程序可能要记住：

- 用户目标是什么
- 当前进行到第几步
- 已经拆出了哪些任务
- 哪些任务还没做
- 哪个步骤失败了

这些“过程中的信息”，就是状态。

---

## 3. 什么是工作流

工作流可以理解成：

`任务从开始到完成所经过的一系列步骤`

例如：

1. 获取用户目标
2. 拆解任务
3. 给每个子任务分类
4. 产出执行计划

如果你把这些步骤固定下来，并且让程序按顺序推进，这就是最基本的 workflow。

---

## 4. 为什么 Day 4 很重要

前面几天你做的示例大多是：

- 提一个问题
- 模型做一次判断
- 返回结果

但真实 Agent 很多时候要处理的是：

- 多步骤任务
- 带中间状态的任务
- 可能失败后重试的任务
- 要根据上一步结果决定下一步的任务

这时候就不能只靠“单轮回答”了。

---

## 5. 今天这个示例会做什么

今天我们做一个最小的“任务拆解工作流”：

- 输入一个目标
- 第一步：模型把目标拆成几个子任务
- 第二步：程序把这些子任务写进状态对象
- 第三步：模型根据当前状态生成下一步执行建议

这个示例重点不是复杂，而是让你看见：

`程序开始维护一个 state 对象，并让后续步骤依赖这个 state。`

## 9. 你运行时重点看什么

重点看这几个输出：

- 初始 state
- task breakdown
- updated state
- next action suggestion

它们分别对应：

- 初始任务状态
- 第一步拆解结果
- 程序写回状态后的结果
- 基于当前状态生成的下一步建议

## 10. 一句话总结

Day 4 的目标不是学新 API，而是建立一个非常关键的 Agent 认知：

`复杂任务一定会经过多个步骤，而多个步骤之间需要共享状态。`
